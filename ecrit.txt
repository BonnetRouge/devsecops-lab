5-Injection SQL
voici le problème :
cursor.execute(f"SELECT * FROM users WHERE name = '{query}'")
Si un utilisateur tape ' OR '1'='1 dans le champ de recherche, la requete devient : 
SELECT * FROM users WHERE name = '' OR '1'='1'
Ce qui retourne tous les utilisateurs de la base! Un attaquant peut ainsi lire, modifier ou supprimer toutes la base de données.

Voici la corection que j'ai apportée
cursor.execute("SELECT * FROM users WHERE name = ?", (query,))
Le ? est un paramettre préparé. Ainsi, Python echappe automatiquement la valeur et l'injection SQL devient impossible.




6-XSS 
Le problème :
return render_template_string(f'<h1>Hello {name}!</h1>')
Si un utilisateur tape <script>alert('hacked')</script> comme nom, le navigateur exécute ce script. Un attaquant peut ainsi voler des cookies de session, rediriger vers un faux site, ou espionner les actions de l'utilisateur.

La correction :

from markupsafe import escape 
return f'<h1>Hello {escape(name)}!</h1>'
escape() transforme les caractères dangereux en texte inoffensif. Par exemple <script> devient &lt;script&gt; — le navigateur l'affiche comme du texte, il ne l'exécute pas.

7-Flash debug et binding

le problème :
app.run(host='0.0.0.0', port=5000, debug=True)
il y a ici, deux problèmes qui se posent : 
-debug=True active le débogueur qui permet d'exécuter du code Python
directement depuis le navigateur. 
En production c'est catastrophique, n'importe qui peut exécuter du code sur ton serveur.
-host='0.0.0.0' : l'application écoute sur toutes les interfaces réseau, donc elle est accessible depuis n'importe quelle machine du réseau, pas seulement en local.

Correction :
app.run(host='127.0.0.1', port=5000, debug=False)
Il y a donc deux corrections :
-debug=False : le débogueur est désactivé en production
-host='127.0.0.1': l'application n'est accessible que localement, depuis la machine elle-même
